# 요구사항 정의

## 핵심 게임 루프

**✅ v1 코어 시스템 재설계 구현 완료 (2026-07-12)** — `GamigotchiStats.mc`(공용 게이지/타이머 모듈) 신설 + `GamigotchiApp.mc`/`GamigotchiBackground.mc`/`GamigotchiDelegate.mc`/`GamigotchiStatusView.mc` 마이그레이션. 시뮬레이터에서 배고픔 감소→아픔→사망, 알→아기→어른 성장 전 구간 자동 검증 완료 (DEVLOG.md 참고). Feed/Play/Clean/Medicine 메뉴 액션과 사망 리셋은 코드 패턴상 안전하지만 버튼 입력이 필요해 아직 실사용자 클릭 검증 전.

```
달리기 → 토큰 획득 → Feed(배고픔 회복) / Play(행복 회복) / Clean(응아 제거) / Medicine(아픈 상태 회복)
                                    ↓
                    배고픔·행복 게이지가 기준치 이상으로 "유지된 시간" 누적
                                    ↓
                          누적 시간이 임계값 도달 → 자동 진화
```

- **성장은 시간 기반으로 전환** (2026-07-12, 기존 "런 횟수 기반"에서 변경). 단, 게이지가 잘 유지된 시간만 누적되므로 간접적으로 달리기(토큰 확보)에 의존 — "달려야 자란다"는 서사는 유지됨
- 달리기는 **토큰**을 획득하는 수단
- 토큰 사용처: Feed(배고픔 회복), Play(행복 회복), Medicine(아픈 상태 회복, 더 비쌈). Clean은 토큰 소모 없음
- 원조 다마고치와 동일한 구조: "언제 진화하느냐"는 시간이 결정, 케어 품질(게이지 유지)이 진화 성공/실패를 좌우

## 코어 스탯 시스템 (v1, 2026-07-12 확정)

### 게이지 2종 (0~100, 시간에 따라 감소)

| 게이지 | 회복 수단 |
|---|---|
| 배고픔 (Hunger) | Feed |
| 행복 (Happiness) | Play |

- 감소 속도, 회복량 등 구체 수치는 밸런스 조정 단계에서 확정 (기존 48h 아픔/72h 사망 감각과 비슷한 스케일로 시작 권장)

### 응아 (Poop)

- 시간마다 주기적으로 쌓임, Clean으로 제거
- 방치 시 두 게이지 감소 속도에 페널티 (수치 미정)

### 아픈 상태 (Sick)

- 배고픔 또는 행복 게이지가 0에 도달하면 즉시 진입
- 회복: **Medicine** 메뉴 액션 — 토큰 소모(Feed보다 비쌈, 예: 3배), 별도 인벤토리 없이 즉시 사용·즉시 회복. 원인이 된 게이지를 정상 범위로 리셋
  - (검토했던 "구매 후 보관하는 진짜 아이템" 방식은 기각 — 새 Storage 스키마/구매-사용 분리/인벤토리 화면까지 필요해 범위가 커짐. v2에서 더 다마고치스럽게 확장하고 싶으면 그때 재검토)

### 사망 (Dead)

- 아픈 상태로 일정 시간(예: 24h) 방치 시 사망
- 기존 구현된 사망 화면/`reviveFromDeath()` 리셋 골격 재사용 가능 (트리거 조건만 교체)

### 성장(진화)

- 두 게이지가 모두 임계값(예: 50) 이상으로 유지된 누적 시간이 기준(예: 알→아기 3일, 아기→어른 10일)을 넘으면 자동 진화
- 게이지가 기준 밑으로 떨어진 동안은 누적이 멈춤(초기화는 안 됨) — 케어가 부실하면 진화가 늦어지는 방식

## 캐릭터

**v1 기본 캐릭터: 펭귄**
- 원래 못 달리는 펭귄이 달리기로 성장한다는 서사
- 귀여움 최우선
- v1 출시 후 반응 좋으면 v2에서 다른 캐릭터 추가

### 캐릭터 디자인 브리프 (2026-07-12 갱신 — "특색 없다" 피드백 반영)

**핵심 결정**: 종(펭귄)은 유지, 대신 스타일화를 강하게. 포켓몬 피플업처럼 "실제 동물을 극단적으로 귀엽게 재해석"하는 방향 — 아예 다른 생물을 만드는 게 아니라 아기 도식(kindchenschema)을 전 단계에 일관되게 적용:

- **눈**: 머리의 30~40%를 차지할 정도로 크게, 반짝이는 하이라이트 점 포함
- **목 생략**: 머리가 몸통에 바로 붙음
- **몸통**: 동그람 우선, 각짐 배제
- **팔다리/부리**: 짧고 뭉툭하게 (가늘고 길면 "우아함"이지 "귀여움"이 아님)
- **전 단계 귀여움 유지 결정**: 청년기를 "날씬하고 꼿꼿한" 성체로 그리지 않기로 함 — 성장한 느낌은 몸매가 아니라 표정/자세의 생기(당당한 눈빛, 팔 벌린 포즈)로 표현. 노년기도 "노쇠함"보다 "정겨운 할아버지" 느낌으로 동글동글함 유지
- **시그니처 요소**: 목에 두르는 **노란색 스카프/헤드밴드**를 전 단계 공통으로 적용 — 달리기 테마와 연결되고, 한눈에 "우리 캐릭터"라고 알아보는 표식. 알 단계는 스카프 대신 알 표면에 노란 무늬로 암시
- **메인 테마 컬러**: 노란색 확정 (스카프/포인트 컬러, 인스타/브랜딩에도 일관 적용 예정)

|------|------|--------|------|
| 알 | 시작 | 펭귄 알 | 흰색 동글동글한 알, 노란 무늬 |
| 아기 | 게이지 유지 누적 시간 기준치 도달 (예: 3일, 추후 조정) | 아기 펭귄 | 회색 솜털, 큰 눈, 노란 스카프 |

각 성장 단계 × 2가지 상태(정상/아픈) = 최소 6종 스프라이트 필요 (v1은 3단계까지만 게임 로직에 연결됨)

| 성장 단계 | 정상 | 아픈 상태 | 게임 로직 연결 |
|-----------|------|-----------|-----------|
| 펭귄 알 | ✅ 필요 | ✅ 필요 | v1 |
| 아기 펭귄(유년기) | ✅ 필요 | ✅ 필요 | v1 |
| 어른 펭귄(청년기) | ✅ 필요 | ✅ 필요 | v1 |
| 노년기 | ✅ 미리 준비 | ✅ 미리 준비 | v2 (아직 미연결) |

- 아픈 상태: 눈이 풀리고 땀 흘리는 버전
- 애니메이션: 캐릭터당 **2프레임** 루프 — ✅ 확정 (2026-07-12). 원조 다마고치도 저프레임으로 충분했다는 근거로 2프레임 유지 결정 (배터리·메모리 부담 최소화)
- 스프라이트 크기: **64×64px** — ✅ 확정 (2026-07-12, 기존 32×32에서 상향). 32×32는 원형 화면에서 너무 작다는 피드백으로 2배 업스케일
- `Toybox.Timer`로 프레임 교체, 손목 들었을 때 애니메이션 시작

### 제작 툴 & 테스트 방법 (2026-07-12 조사)

- **AI 생성 (메인 경로)**: [PixelLab.ai](https://www.pixellab.ai/) — MCP 서버 지원으로 Claude Code에서 직접 `create_character`/`animate_character` 호출 가능 (`claude mcp add pixellab https://api.pixellab.ai/mcp -t http -H "Authorization: Bearer $TOKEN"`). 청년기 기준 캐릭터를 먼저 생성 후 reference로 유년기/노년기 변형하는 순서로 스타일 일관성 확보
- **수동 제작 (대안)**: [Piskel](https://www.piskelapp.com/) (무료, 웹 기반, 애니메이션 스프라이트 전용) 또는 Aseprite(유료, 더 전문적)
- **AI 이미지 → 스프라이트 변환**: `tools/ai_to_sprite.py` — 임의 크기/색상 이미지를 64×64로 다운스케일 + 64색 팔레트 스냅 + 투명 배경 처리까지 자동화. 이미 픽셀아트인 소스는 NEAREST 다운스케일 사용(BOX/평균은 경계에 의도치 않은 색 얼룩 생김). PixelLab처럼 이미 규격에 맞게 나오는 도구를 안 쓰는 경우(범용 AI 이미지 생성 등)에 사용
- **색상 팔레트**: 가민 MIP 디스플레이는 RGB 각 채널이 `0x00/0x55/0xAA/0xFF` 4단계 조합인 64색 고정 팔레트. 이 밖의 색은 디더링(도트 패턴)으로 뭉개짐. `resources/palette-64.png`에 64색 스와치 이미지 준비해둠
- **디더링 방지**: `resources/drawables/drawables.xml`의 각 `<bitmap>`에 `dithering="none"` 이미 적용됨. 팔레트 안 색만 쓰면 시뮬레이터/실기기에서 그린 그대로 나옴
- **파일명 규칙**: `resources/drawables/sprites/{egg|baby|adult}_{normal|sick}_{1|2}.png` — 지금 placeholder 단색 이미지가 들어있는 자리에 같은 이름으로 덮어쓰면 코드 수정 없이 바로 반영됨. 노년기는 `elder_{normal|sick}_{1|2}.png`로 미리 준비만 해두고 v2에서 연결
- **테스트 루프**: monkeyc로 빌드 → 시뮬레이터에 푸시 → 스크린샷으로 원형 베젤 안에서 실제로 어떻게 보이는지 확인하는 파이프라인 구축됨 (DEVLOG.md 참고). 이미지 교체 후 이 루프로 바로 검증 가능

## 화면 구성

대상 기기: Forerunner 255 (260×260px 원형, MIP 디스플레이 64색)

- ~~현재 시간: 상단~~ — ✅ 제거 확정 (2026-07-12). 캐릭터를 크게 보여주는 데 집중, 시간 확인은 워치 본래 기능이므로 앱 안에서는 생략. **트레이드오프**: 앱이 열려있는 동안 시간을 못 봄 (장치 앱이 워치페이스를 대체하는 특성상)

- 말풍선: 캐릭터 옆 또는 위, 현재 상태 표시
- 보유 토큰: 하단
- 손목을 들었을 때 2~3초 애니메이션 재생

### 말풍선 상태 메시지

**⚠️ 한글 미지원**: Connect IQ 기본 시스템 폰트가 한글(Hangul)을 렌더링 못 함 (시뮬레이터에서 물음표 다이아몬드로 깨지는 것 확인, 2026-07-11). 커스텀 폰트 리소스를 임베드하지 않는 한 모든 화면 텍스트는 영어로 작성. 아래 표의 문구는 실제 구현 시 영어로 대체됨 (예: "no tokens...", "yum yum!").

| 조건 | 말풍선 (기획 의도, 실제는 영어로 구현) |
|---|---|
| 정상 (최근 급식) | 없음 (애니메이션만) |
| 마지막 급식 후 36시간 경과 | "배고파요..." → `hungry...` |
| 아픈 상태 (48시간 초과) | "너무 배고파요 😢" → `so hungry...` |
| 밥 먹은 직후 (잠깐 표시) | "냠냠 🐟" → `yum yum!` ✅ 구현됨 |

v2 추가 예정:
- 응아 쌓임 → "치워줘요 💩"
- 관심 요청 → "놀아줘요!"

## 버튼 레이아웃

```
[왼쪽 상단 = (미사용, 백라이트 전용)]  [오른쪽 상단 = 메뉴 열기 / 선택 확인]
[왼쪽 중단 = 상태 확인          ]
[왼쪽 하단 = (미사용)           ]  [오른쪽 하단 = 뒤로 / 닫기]
```

- 왼쪽 상단(LIGHT) 버튼은 실기기/시뮬레이터 모두 백라이트 전용으로 시스템이 예약해서 앱에 이벤트가 전달되지 않음 (2026-07-11 시뮬레이터 테스트로 확인) → 상태 확인 화면은 왼쪽 중단(Up) 버튼으로 이동

### 메뉴 플로우 (오른쪽 상단)

```
메인 화면
  → 오른쪽 상단: 메뉴 열기
    → [ Feed (밥주기)      ]  (v1)
      [ Play (놀아주기)    ]  (v1, 2026-07-12: v2→v1로 승격 — 게이지 시스템 도입에 따라 필수 액션이 됨)
      [ Clean (응아 치우기) ]  (v1, 위와 동일 사유로 승격)
      [ Medicine (약 주기)  ]  (v1, 게이지 시스템 신규 추가 — 아픈 상태 회복용)
  → 왼쪽 중/하단: 항목 이동
  → 오른쪽 상단: 선택 확인
  → 오른쪽 하단: 취소 / 닫기
```

### 왼쪽 상단 — 상태 확인 화면

- 보유 토큰, 배고픔/행복 게이지, 진화까지 남은 유지 시간 표시 (2026-07-12: "런 횟수" 대신 게이지 기반으로 변경)
- 한 번 더 누르거나 오른쪽 하단으로 메인 화면 복귀

### v1 (MVP) — 2026-07-12 게이지 시스템 도입으로 범위 갱신
- 런 중 거리 읽기 → 토큰 변환 → Storage 저장
- 배고픔/행복 게이지 시간 경과 감소 + Feed/Play로 회복
- 응아 주기적 발생 + Clean으로 제거

- 아픈 상태(게이지 0) → Medicine으로 회복 / 방치 시 사망(리셋)
- 장치 앱 기본 화면 (시간 + 캐릭터 + 토큰)
- 상태 확인 화면 (토큰, 게이지, 진화 진행도)
- **진화 축하 연출** — ✅ 구현 완료 (2026-07-12). 성장 단계가 오를 때 방사형 광선 이펙트(그래픽 도형만 사용, 이미지 리소스 불필요) + 새 단계 스프라이트 + "Evolved into {Stage}!" 문구를 3.5초간 표시. `GamigotchiStats._accumulateGrowth()`에서 `pendingEvolution` 플래그를 세팅하고, 다음 앱 실행 시 `GamigotchiView.onShow()`가 감지해서 표시

### 공유/바이럴 전략 (2026-07-12 논의)

목표를 "달리기 동기부여"에서 "가미고치 자체가 재밌어서 기르고 공유"로 확장. 달리기는 그대로 "입력 엔진"으로 유지하되, 결과물(개체)의 차별성과 공유 욕구를 자극하는 요소에 투자하는 방향으로 정리:

- **진화 축하 연출** (위 참고) — 공유하고 싶은 "순간"을 만드는 첫 번째 장치로 구현
- **카메라/AR 합성** (사진 찍을 때 캐릭터가 같이 나오는 기능) — ❌ 이 프로젝트(워치 장치 앱) 범위 밖. 가민 워치에 카메라가 없어서 별도 폰 컴패니언 앱(Connect IQ Mobile SDK)이 필요한 완전히 다른 규모의 프로젝트. 보류
- **성장 일지/타임라인 화면** (돌보는 과정을 쉽게 볼 수 있게) — 아직 미구현, 제안 단계. 각 진화 시점의 타임스탬프를 기록해뒀다가 이력으로 보여주는 화면. 원하면 다음에 작업

### v2 (업데이트)
- 꾸미기 아이템 시스템 (토큰 소모)
- **체중/운동 밸런스** — 원조 다마고치의 "간식 과다 → 체중 증가 → 운동으로 관리" 메커니즘을 가마고치 컨셉에 맞게 재해석. 꾸미기용 간식 아이템을 많이 주면 체중 증가, 달리기(실제 러닝 거리)로만 다시 빠짐. "실제로 달려야 관리된다"는 핵심 서사를 강화하는 방향. 별도 스탯(`weight` 등) 및 화면 반영(체형 변화 스프라이트) 설계 필요
- 진화 분기 (케어 품질에 따른 다른 어른 펭귄) — 분기를 결정할 케어미스 카운터(배고픈 채로 방치한 횟수 등) 구체 설계 필요. **분기 수는 2개(잘 돌봄/방치함)로 제한 권장** — 분기마다 별도 스프라이트가 필요해서 아트 비용이 분기 수에 비례해 늘어남
- 캐릭터 추가 (펭귄 외)
- 멀티 기기 지원

## 기술 제약 및 확인 필요 사항

### 확인 완료 ✅

- **런 중 거리 읽기** — `Activity.getActivityInfo().elapsedDistance`로 현재 달리기 거리(미터) 실시간 읽기 가능
  - 누적 히스토리 API는 없음 → 앱이 열려 있는 동안 거리를 읽고 직접 저장하는 방식으로 구현
- **데이터 저장** — `Application.Storage.setValue/getValue`로 영구 저장 가능 (앱 재시작 후에도 유지)
  - 저장 가능 타입: Number, Float, String, Boolean, Array, Dictionary 등
  - 값당 최대 32KB 제한
- **토큰 계산** — API 문제 아님. 저장된 거리를 수식으로 토큰으로 변환하면 됨 (예: 1km = N토큰)

### 핵심 데이터 흐름

```
앱 열림 (런 중) → elapsedDistance 읽기 → 런 종료 시 거리 저장
→ 거리 × 토큰 공식 = 토큰 획득 → Storage에 누적 저장
```

### Storage 데이터 구조 (게이지 시스템, 2026-07-12 구현 완료)

| 키 | 타입 | 설명 |
|---|---|---|
| `tokens` | Number | 보유 토큰 수 |
| `growthStage` | Number | 0=알, 1=아기, 2=어른 |
| `healthStatus` | Number | 0=정상, 1=아픔, 2=사망 |
| `hunger` | Float | 배고픔 게이지 (0~100) |
| `happiness` | Float | 행복 게이지 (0~100) |
| `poopCount` | Number | 쌓인 응아 개수 |
| `poopAccumSeconds` | Number | 다음 응아 발생까지 누적된 시간 |
| `healthyElapsedSeconds` | Number | 두 게이지가 기준치 이상으로 유지된 누적 시간(성장 타이머) |
| `lastTickTime` | Number | 게이지 감소량 계산용 마지막 업데이트 시각 |
| `sickSinceTime` | Number | 아픈 상태 진입 시각 (사망 판정용) |
| `hungerNotified` | Boolean | 배고픔 알림 중복 발송 방지 플래그 |

`qualifyingRunCount`, `lastFedTime`은 게이지 시스템 도입으로 폐기됨.

### 필수 확인 (개발 시작 전)

- **[확인 완료 ✅] 백그라운드 런 감지** — `Background.registerForActivityCompletedEvent()`로 앱이 닫혀 있어도 런 종료 이벤트 수신 가능
  - `ServiceDelegate`를 상속한 백그라운드 서비스 클래스 구현 → `onActivityCompleted()` 메서드에서 거리 읽기 → 토큰 계산 → `Storage` 저장 → `Background.exit(data)`로 메인 앱에 전달
  - 메인 앱 열릴 때 `AppBase.onBackgroundData()`로 백그라운드 결과 수신
  - 관련 API: `Toybox.Background`, `Toybox.Activity`
- **[확인 완료 ✅] 알림 기능** — `Toybox.Notifications.showNotification()`으로 즉시 알림 표시 가능. 단, 예약 발송 기능은 없음
  - 구현 방법: 밥줄 때마다 `Background.registerForTemporalEvent(48시간 후)`로 재등록 → 시간 됐을 때 백그라운드에서 마지막 급식 시각 확인 → 조건 충족 시 `showNotification()` 호출
  - `Notifications` 퍼미션 선언 필요 (manifest.xml)
  - 주의: 앱 종류에 따라 지원 기기가 다를 수 있음. Forerunner 255 지원 여부 실기기 확인 필요

- **[확인 완료 ✅] 워치페이스 연동 불가** — Connect IQ 앱은 각자 독립된 Storage를 가지며 앱 간 데이터 공유 없음
  - 워치페이스에서 장치 앱의 Storage를 읽는 것 불가
  - **결론**: 장치 앱 단독으로 구현하는 현재 방향이 맞음. 워치페이스 연동은 고려 불필요
  - **대안으로 고려 가능**: Glance(글랜스) — 버튼 한 번으로 접근하는 미니 뷰. 별도 앱이지만 장치 앱과 함께 번들 배포 가능한지 추가 확인 필요 (v2 검토 항목)

## 미확정 항목 (추가 논의 필요)

### 게임 플로우

- **사망 플로우** — ✅ 확정 (2026-07-11). 72시간 미급식 시 `healthStatus=2`(사망)로 전환, 별도 사망 화면("Your pet has died." + 묘비 스프라이트) 표시. 자동 리셋 안 하고 SELECT 버튼을 눌러야 새 알로 리셋 (`GamigotchiApp.reviveFromDeath()`). 시뮬레이터에서 동작 확인 완료.
  - **(v2 추가 결정, 2026-07-12)** 노년기 도입 시, 노년기 도달 후 일정 기간이 지나면 수명으로 자연사하는 것도 추가하기로 함. "방치로 인한 사망(72h)"과는 별개 트리거 — 이전에 논의했던 "성실한 유저를 나이로 벌주면 안 된다"는 원칙과는, 노년기까지 도달한 유저는 이미 오랫동안 잘 돌본 유저라는 점에서 상충하지 않는다고 판단(오히려 "한 생애를 완주했다"는 서사로 활용 가능 — 다마고치의 세대 교체 컨셉과 유사). 구체 기간/연출은 노년기 단계 설계 시 확정
- **첫 시작** — ✅ 확정 (2026-07-11). 별도 인트로 화면 없이, 최초 설치 시 알과 함께 말풍선으로 "take care of your egg!" 잠깐 표시 (기존 말풍선 메커니즘 재사용, `_transientMessage`).

### 엣지 케이스

- **토큰 부족 시 밥주기 시도** — ✅ 해결. 말풍선 "no tokens..." 5초간 표시 (`GamigotchiApp.feed()`)
- **밥주기/놀아주기/약 소모 토큰** — ✅ 구현 완료 (2026-07-12, 수치는 가안). Feed 1토큰(배고픔+40), Play 1토큰(행복+40), Medicine 3토큰(두 게이지 모두 건강 임계값 50까지 회복), Clean은 토큰 소모 없음. `GamigotchiApp.mc`의 `FEED_COST`/`PLAY_COST`/`MEDICINE_COST` 상수 — 밸런스 조정 단계에서 재조정 가능

### 알림

- **배고픔 알림 타이밍** — ✅ 구현 완료 (2026-07-12). 배고픔 또는 행복 게이지가 **30 이하**로 떨어지면 1회 알림 (`GamigotchiBackground.ALERT_THRESHOLD`), 게이지가 30 초과로 회복되면 알림 플래그가 풀려 다음 위기 때 다시 알림 가능
  - `Toybox.Notifications.showNotification()`이 `(:background)` 컨텍스트에서 크래시나서(`Error: Unexpected Type Error — Failed invoking <symbol>`, 시뮬레이터 확인, Garmin 포럼에도 유사 사례) `Background.requestApplicationWake()`로 구현
