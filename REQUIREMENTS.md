# 요구사항 정의

## 핵심 게임 루프

**✅ v1 코어 시스템 재설계 구현 완료 (2026-07-12)** — `GamigotchiStats.mc`(공용 게이지/타이머 모듈) 신설 + `GamigotchiApp.mc`/`GamigotchiBackground.mc`/`GamigotchiDelegate.mc`/`GamigotchiStatusView.mc` 마이그레이션. 시뮬레이터에서 배고픔 감소→아픔→사망, 알→아기→어른 성장 전 구간 자동 검증 완료 (DEVLOG.md 참고). Feed/Play/Clean/Medicine 메뉴 액션과 사망 리셋은 코드 패턴상 안전하지만 버튼 입력이 필요해 아직 실사용자 클릭 검증 전.

```
달리기 → 토큰 획득 → Feed(배고픔 회복) / Play(행복 회복) / Clean(응아 제거) / Medicine(아픈 상태 회복)
                                    ↓
                    배고픔·행복 게이지가 기준치 이상으로 "유지된 시간" 누적
                                    ↓
                          누적 시간이 임계값 도달 → 자동 진화
```

- **성장은 시간 기반으로 전환** (2026-07-12, 기존 "런 횟수 기반"에서 변경). 단, 게이지가 잘 유지된 시간만 누적되므로 간접적으로 달리기(토큰 확보)에 의존 — "달려야 자란다"는 서사는 유지됨
- 달리기는 **토큰**을 획득하는 수단
- 토큰 사용처: Feed(배고픔 회복), Medicine(아픈 상태 회복, 더 비쌈). ~~Play~~ Clean은 토큰 소모 없음 (2026-07-13 변경: Play는 토큰 대신 배고픔 소모 — "행복 스탯 역할 분리" 참고)
- 원조 다마고치와 동일한 구조: "언제 진화하느냐"는 시간이 결정, 케어 품질(게이지 유지)이 진화 성공/실패를 좌우

### 달리기 ↔ 성장 관계 철학 (2026-07-13 확정)

- **달리기 = 성장의 "연료", 케어 = 성장의 "조건".** 달린 거리가 성장을 직접 가속하는 방식은 **명시적으로 반대** — "많이 달린 사람이 빨리 큰다"가 아님
- 대신: 달리기 → 토큰 → 밥. 놀이는 배고픔을 소모하므로 잘 놀아줄수록 밥(=토큰=달리기)이 더 필요 → **달리기 없이는 케어 자체가 안 돌아감.** 즉 달리기는 "점수"가 아니라 **케어를 돌리는 필수 인프라**
- 결과적으로 "꾸준히 달려서 잘 먹이고 잘 놀아주며 관리하면 성장"하는 그림 — 거리 경쟁이 아니라 **꾸준함**이 핵심

## 코어 스탯 시스템 (v1, 2026-07-12 확정)

### 게이지 2종 (0~100, 시간에 따라 감소)

| 게이지 | 회복 수단 | 성격 | 실패 시 |
|---|---|---|---|
| 배고픔 (Hunger) | Feed (토큰 소모) | 생존·성취 축 (유능감) | **하드 — 0 방치 시 아픔→사망** |
| 행복 (Happiness) | Play (토큰 X, 아래 참고) | 관계·애착 축 (관계성) | **소프트 — 안 죽음** |

- 감소 속도, 회복량 등 구체 수치는 밸런스 조정 단계에서 확정 (기존 48h 아픔/72h 사망 감각과 비슷한 스케일로 시작 권장)
- ⚠️ **초기 구현은 두 게이지가 완전 대칭이었으나(둘 다 토큰 소모·둘 다 사망 트리거), 2026-07-13 아래 설계로 역할을 분리하기로 확정** — 대칭이면 스탯이 2개일 이유가 없고, "안 놀아줬다고 죽는다"는 손실 회피가 과해 이탈 유발(심리 설계 line 215)

### 배고픔 vs 행복 역할 분리 (2026-07-13 확정)

**배고픔 = 달리기 엔진.** 컨셉 확정: 배가 고프면 → 실제로 달려서 모은 토큰으로 → 밥을 먹여 올린다. 앱의 근본 동기부여(러닝) 루프를 담당.

**행복 = 관계성(애착) 축.** 배고픔의 복사본이 되지 않도록 아래로 분리:

- **올리는 법 — Play는 토큰을 안 쓰는 대신 배고픔을 소량 소모**한다. 놀아주면 행복↑ + 배고픔↓ → 밥이 필요해짐 → 결국 달리기로 되돌아옴. 토큰을 안 쓰면서도 러닝 엔진과 자연스럽게 결합되는 구조. (수치: Play → 행복 +40, 배고픔 -5 — 2026-07-14에 -10→-5로 완화)
- **실패는 소프트** — 행복 0으로는 **죽지 않는다**. 대신 시무룩한 표정 + 웅크림(활동성↓) + 성장 정지 + 긍정 서프라이즈(방향 B) 차단.
- **행복이 표정·활동성을 파생** (투자가 화면에 보이게 = 매몰비용 가시화, line 212):

  | 행복 | 표정 | 활동성 (걷기 상태 머신) |
  |---|---|---|
  | 높음 (70+) | 활짝 | 자주·넓게 배회, 서있는 시간 짧음, 특별 모션 확률↑ |
  | 보통 (30~70) | 평범 | 현재 기본값 |
  | 낮음 (<30) | 시무룩 | 거의 안 움직이고 웅크림, 서있는 시간 김 |

  - **활동성**(걷기 빈도·범위)은 기존 걷기 프레임 재사용 → **새 아트 0, 코드만**으로 즉시 구현 가능
  - **표정**(활짝/시무룩)만 방향 D 아트 필요 — 눈·입 몇 픽셀이라 코드 오버레이 무료 시도 → 안 예쁘면 PixelLab

- **놀아주는 장면**: Play 시 짧은 연출.
  - 무료 버전(즉시 가능): 포포가 콩콩 뛰고(기존 프레임 상하 바운스) + 하트/별 파티클(진화 축하와 같은 코드 도형) + 말풍선
  - 업그레이드(나중·PixelLab): 전용 "노는 포포" 스프라이트
  - 방향 B(가변 보상) 합류 지점 — 놀아줄 때 낮은 확률로 특별 리액션/모션이 열림
  - ⚠️ **다음 작업으로 확정 (2026-07-14)**: 현재는 텍스트 말풍선("wheee!")만 있어 Play가 배고픔을 깎는 게 "노는데 왜 생명력이 깎이지?"로 느껴진다는 실기기 테스트 피드백 발생. 콩콩 뛰기+파티클 애니메이션을 넣어 수치 변화를 자연스러운 연출로 감싸는 게 다음 작업

### 응아 (Poop)

- 시간마다 주기적으로 쌓임, Clean으로 제거
- 방치 시 두 게이지 감소 속도에 페널티 (수치 미정)

### 아픈 상태 (Sick)

- **배고픔** 게이지가 0에 도달하면 즉시 진입 → 방치 시 사망 (하드 실패)
- 행복 0은 아픔/사망으로 이어지지 않음 — 소프트 페널티만 (위 "역할 분리" 참고). ⚠️ 초기 구현은 행복 0도 아픔→사망으로 갔으나 2026-07-13 변경 확정
- (구현 메모: 현재 `GamigotchiStats._handleCriticalGauge`는 `hunger<=0 || happiness<=0` 양쪽을 트리거로 봄 → 배고픔 단독 트리거로 수정 필요)
- 회복: **Medicine** 메뉴 액션 — 토큰 소모(Feed보다 비쌈, 예: 3배), 별도 인벤토리 없이 즉시 사용·즉시 회복. 원인이 된 게이지를 정상 범위로 리셋
  - (검토했던 "구매 후 보관하는 진짜 아이템" 방식은 기각 — 새 Storage 스키마/구매-사용 분리/인벤토리 화면까지 필요해 범위가 커짐. v2에서 더 다마고치스럽게 확장하고 싶으면 그때 재검토)

### 사망 (Dead)

- 아픈 상태로 일정 시간(예: 24h) 방치 시 사망
- 기존 구현된 사망 화면/`reviveFromDeath()` 리셋 골격 재사용 가능 (트리거 조건만 교체)

### 성장(진화)

- 두 게이지가 모두 임계값(예: 50) 이상으로 유지된 누적 시간이 기준(예: 알→아기 2시간, 아기→어른 10일)을 넘으면 자동 진화
- 게이지가 기준 밑으로 떨어진 동안은 누적이 멈춤(초기화는 안 됨) — 케어가 부실하면 진화가 늦어지는 방식
- ✅ **구현 불일치 해결됨** (2026-07-15): `_accumulateGrowth`가 이제 두 게이지 모두 `HEALTHY_THRESHOLD` 이상일 때만 누적하도록 수정 완료

**단계별 케어 차등 (2026-07-13 확정)** — 단계마다 빨리 닳는 게이지를 다르게 해서 생애 주기별로 신경 쓸 게 달라지게. 기준 감소는 `100/48 ≈ 2.08/시간`, 여기에 단계별 배율:

| 단계 | 배고픔 감소 배율 | 행복 감소 배율 | 케어 초점 |
|---|---|---|---|
| 알 | ×0.4 (느림) | ×0.4 (느림) | 부화 대기 — 케어 부담 최소 |
| 유아기 | **×1.6** | ×0.7 | **자주 먹여야** |
| 청년기 | ×0.8 | **×1.6** | **잘 놀아줘야** |

- 배율 수치는 시작점, 실기기 테스트에서 조정
- **알 단계 = 짧은 기대감 비트로 확정** (2026-07-13): 3일은 너무 지루 → 알은 짧게(몇 시간, 튜닝) + 케어 거의 불필요 + **부화 축하 연출** 후 유아기부터 본격 육아. 탄생 모먼트/기존 알 아트·이스터에그를 살리는 선택
  - **✅ 2026-07-15 반영**: `STAGE_THRESHOLDS_SEC[0]`을 3일→**2시간**으로 수정(가안, 실기기 테스트로 조정). 현행 단일 감소율(≈2.08/시간)로도 게이지 100→50에 약 24시간이 걸려 2시간 부화엔 케어 불필요 — "케어 거의 불필요" 의도와 일치. ⚠️ 단, 위 표의 단계별 감소 배율(알 ×0.4 등)은 **아직 코드에 미구현**(설계 감사 #11 참고)
  - **부화 놓침 걱정 없음**: 기존 `pendingEvolution` 플래그 방식 그대로 — 앱이 닫힌 동안 조건 충족돼도 다음에 앱 열 때 `onShow()`가 연출 재생. 구조상 놓칠 수 없음
  - (옵션·나중) "알을 탭해서 따뜻하게 → 부화" 인터랙션을 얹으면 첫 순간부터 능동 관여 가능. v1은 자동 부화로 시작

### 캐릭터 디자인 브리프 (2026-07-12 갱신 — "특색 없다" 피드백 반영)

**핵심 결정**: 종(펭귄)은 유지, 대신 스타일화를 강하게. 포켓몬 피플업처럼 "실제 동물을 극단적으로 귀엽게 재해석"하는 방향 — 아예 다른 생물을 만드는 게 아니라 아기 도식(kindchenschema)을 전 단계에 일관되게 적용:

- **눈**: 머리의 30~40%를 차지할 정도로 크게, 반짝이는 하이라이트 점 포함
- **목 생략**: 머리가 몸통에 바로 붙음
- **몸통**: 동그람 우선, 각짐 배제
- **팔다리/부리**: 짧고 뭉툭하게 (가늘고 길면 "우아함"이지 "귀여움"이 아님)
- **전 단계 귀여움 유지 결정**: 청년기를 "날씬하고 꼿꼿한" 성체로 그리지 않기로 함 — 성장한 느낌은 몸매가 아니라 표정/자세의 생기(당당한 눈빛, 팔 벌린 포즈)로 표현. 노년기도 "노쇠함"보다 "정겨운 할아버지" 느낌으로 동글동글함 유지
- **시그니처 요소**: 목에 두르는 **노란색 스카프/헤드밴드**를 전 단계 공통으로 적용 — 달리기 테마와 연결되고, 한눈에 "우리 캐릭터"라고 알아보는 표식. 알 단계는 스카프 대신 알 표면에 노란 무늬로 암시
- **메인 테마 컬러**: 노란색 확정 (스카프/포인트 컬러, 인스타/브랜딩에도 일관 적용 예정)

## 성장 단계

**v1은 3단계 유지** (2026-07-12 재확인). "노년기" 4단계 확장은 v2로 미룸 — 단, 아래 픽셀 아트는 노년기까지 미리 준비해둠 (PixelLab 생성 작업을 한 번에 처리하기 위함이고, 게임 로직에는 아직 연결 안 함).

| 단계 | 조건 (2026-07-12 변경: 게이지 유지 시간 기반) | 케어 초점 (2026-07-13) | 캐릭터 | 모습 |
|------|------|------|--------|------|
| 알 | 시작 | 부화 대기 (케어 최소, 짧게) | 펭귄 알 | 흰색 동글동글한 알, 노란 무늬 |

### 토큰 상한 (2026-07-13 확정) — 이중 상한

무한정 쌓이면 (1) 잉여 토큰이 쓸 데 없이 남고 (2) 화면에 표시할 공간이 부족(자릿수 폭주). 그래서 두 상한을 함께 둠:

- **지갑 보유 한도** (예: 30토큰) — 총 보유 상한. 며칠치 저축은 되지만 무한 파밍 차단 + 화면 표시 2자리 유지

- ⚠️ 상한 설계 시 원칙: "동기부여 앱에서 상한을 채우면 안 뛰게 되는" 함정을 피하려 **지갑 상한보다 하루 리셋 구조를 러닝 동기의 축으로** 삼음

### 제작 툴 & 테스트 방법 (2026-07-12 조사)

- **AI 생성 (메인 경로)**: [PixelLab.ai](https://www.pixellab.ai/) — MCP 서버 지원으로 Claude Code에서 직접 `create_character`/`animate_character` 호출 가능 (`claude mcp add pixellab https://api.pixellab.ai/mcp -t http -H "Authorization: Bearer $TOKEN"`). 청년기 기준 캐릭터를 먼저 생성 후 reference로 유년기/노년기 변형하는 순서로 스타일 일관성 확보

- **AI 이미지 → 스프라이트 변환**: `tools/ai_to_sprite.py` — 임의 크기/색상 이미지를 64×64로 다운스케일 + 64색 팔레트 스냅 + 투명 배경 처리까지 자동화. 이미 픽셀아트인 소스는 NEAREST 다운스케일 사용(BOX/평균은 경계에 의도치 않은 색 얼룩 생김). PixelLab처럼 이미 규격에 맞게 나오는 도구를 안 쓰는 경우(범용 AI 이미지 생성 등)에 사용
- **색상 팔레트**: 가민 MIP 디스플레이는 RGB 각 채널이 `0x00/0x55/0xAA/0xFF` 4단계 조합인 64색 고정 팔레트. 이 밖의 색은 디더링(도트 패턴)으로 뭉개짐. `resources/palette-64.png`에 64색 스와치 이미지 준비해둠
- **디더링 방지**: `resources/drawables/drawables.xml`의 각 `<bitmap>`에 `dithering="none"` 이미 적용됨. 팔레트 안 색만 쓰면 시뮬레이터/실기기에서 그린 그대로 나옴
- **파일명 규칙**: `resources/drawables/sprites/{egg|baby|adult}_{normal|sick}_{1|2}.png` — 지금 placeholder 단색 이미지가 들어있는 자리에 같은 이름으로 덮어쓰면 코드 수정 없이 바로 반영됨. 노년기는 `elder_{normal|sick}_{1|2}.png`로 미리 준비만 해두고 v2에서 연결
- **테스트 루프**: monkeyc로 빌드 → 시뮬레이터에 푸시 → 스크린샷으로 원형 베젤 안에서 실제로 어떻게 보이는지 확인하는 파이프라인 구축됨 (DEVLOG.md 참고). 이미지 교체 후 이 루프로 바로 검증 가능

**⚠️ 핵심 원칙 (문서 8번)**: 손실 회피와 가변적 보상은 과용 시 불안·강박을 유발. 특히 **달리기 동기부여 앱에서 과한 페널티는 "더 달리기"가 아니라 "앱 삭제(이탈)"로 이어짐.** 따라서 부정 기제는 순화된 형태로만 도입.

### 도출된 4개 실행 방향 (제안 단계, 우선순위순)

**방향 B — 긍정적 가변 보상 (최우선, 저비용·즉효)**
- 런 종료 시 낮은 확률 보너스 토큰(예: 10%×2), 손목 들 때 가끔 특별 모션(하품·두리번·스카프 고쳐매기 등, 예: 15%), 케어 시 낮은 확률 특별 리액션
- 전부 **"좋은 쪽" 서프라이즈만**. 부정적 랜덤(토큰 깎임 등) 절대 배제
- 구현 지점: `GamigotchiBackground.onActivityCompleted()` 토큰 계산 + 손목 들 때 애니메이션 선택 로직

**방향 D — 얼굴/표정 강조**
- 현재 스프라이트 상태는 정상/아픔 2종뿐 → 게이지에서 자동 파생되는 표정 3~4종 추가(배고픔 낮음→시무룩, 둘 다 높음→활짝, 방금 먹음→하트눈)
- 근거: 문서 4번 ELIZA 효과(최소 신호로 애착) + 원칙 "딱 보여야 함"·"감정 연결" + 디자인 브리프 "눈 30~40%"(line 68). 화면 128px로 확대돼 얼굴에 픽셀 배분 여유 있음
- ⚠️ 트레이드오프: 표정 상태 수만큼 스프라이트 증가 → 게이지 파생 3~4종으로 시작
- **✅ 코드 레이어 구현 완료 (2026-07-15)** — `GamigotchiStats.computeExpression(hunger, happiness)`로 시무룩/활짝/평범 계산, `GamigotchiApp.getExpression()`이 급식 직후 하트눈(5초, 말풍선과 동일 길이)을 우선 적용. `GamigotchiView._get{Baby,Adult}ExpressionBitmapId()`가 표정별 분기를 갖고 있으나 **전용 아트가 없어 전부 기존 Normal 스프라이트로 폴백** — 아트 완성 시 주석 해제 + `drawables.xml` 등록만 하면 반영됨
- **아트 획득 방법 조사**: 기존 normal 스프라이트(128×128)에 이미 눈이 그려져 있어(어른/아기 모두 실제 완성 아트) 코드로 눈만 오버레이해 바꿔치기하는 건 위험 — 검은 픽셀 자동 탐지 시도 결과 턱시도(검은 몸통)와 눈 색이 겹쳐 구분 불가 확인(2026-07-15). 두 갈래로 압축:
  1. **Piskel 손수정(크레딧 0)** — 기존 PNG 열어서 눈/입 부분만 수동 교체, 몸통/포즈 그대로라 어긋날 위험 없음. 표정 3종 정도는 몇 분이면 될 만큼 간단
  2. **PixelLab 레퍼런스 변형(크레딧 소모)** — "이 캐릭터 그대로 눈만 다르게" 요청. 일관성은 낫지만 표정(3)×단계(2)×프레임(2)=최대 12장이라 크레딧 부담
  - 몸통/포즈가 그대로라 방향 E 리액션 포즈만큼 비싸진 않음. **Piskel 먼저 시도 권장**, 안 예쁘면 PixelLab로 전환
- ⚠️ **제안 (2026-07-14, 실기기 테스트 중 논의)**: 메인 화면의 하트/별 숫자 게이지(line 202)를 없애고 표정+말풍선만으로 상태를 암시하는 쪽으로 전환 검토. 원조 다마고치는 캐릭터만 보여주고 수치를 노출하지 않음 — 지금처럼 게이지 막대가 그대로 떠 있으면 "다마고치라기보다 대시보드"에 가까운 느낌. **절충안**: 메인 화면은 표정/애니메이션으로만 암시하고, Up 버튼으로 여는 Status 화면엔 정확한 수치를 남겨서(지금 구조 그대로) 원하면 확인 가능하게 유지 — 완전히 수치를 숨기면 "언제 케어해야 하는지" 판단이 어려워져 손실 회피 순화 원칙과 상충할 수 있음. 방향 D 아트(표정 스프라이트) 완료가 선행 조건

**방향 C — 개체 이름 붙이기**
- 이름의 효과는 "다양성"이 아니라 **개별화** — 캐릭터가 하나여도 "그냥 펭귄"→"내 펭귄 포포"로 애착 급상승
- ⚠️ 한글 폰트 미지원(line 144) → 영문 입력 또는 프리셋 이름 선택으로 우회
- 캐릭터 다양성은 이름의 전제조건이 아님. 다양성은 방향 A(세대교체)를 통해 아트 비용 최소로 따라옴

**방향 A — 사망 → "완주 서사" 재설계 (설계 규모 큼)**
- 문제: 현재 사망은 성장·토큰을 **완전 제로로** 리셋 → 손실 회피가 과해 이탈 유발
- 재설계: 죽음을 "실패, 0부터 다시"가 아니라 **"한 생애 완주, 이야기는 계속"**(다마고치 세대교체 컨셉, line 320)으로 리프레이밍
  1. **세대 카운터 + 기록** — Storage에 `generation` + 죽은 개체 기록(도달 단계·함께 달린 총 거리·돌본 일수). 리셋=소멸이 아니라 축적(가족사진첩). 순수 손실 → 매몰비용(긍정)+유능감으로 변환
  2. **부분 이월** — 토큰 일부 이월 또는 다음 알에 "경험자 표식"(스카프 색 등). 절벽 제거로 이탈 방지
  3. **두 죽음 구분** — 방치사(72h)는 부드러운 페널티, 자연사(v2 노년기 수명)는 완전한 축하 연출("좋은 삶을 살았어요, N대 시작")
- **허브 역할**: 세대마다 이름(C) + 세대마다 살짝 다른 개체 → **다양성이 공짜로** 생김. 이름·다양성·손실 순화·매몰비용을 한 번에 엮음

**방향 E — 런 데이터 리액션 (관찰의 재미, 2026-07-15 확정)**
- **문제의식**: 방향 B~D는 심리 기제(가변보상/표정/이름) 축이지만, "미니게임 없이도 재밌나"라는 근본 질문에 대한 답은 **"바라보면서 성장하는" 컨셉 자체를 강화**하는 데 있음 — 내가 실제로 뛴 게 캐릭터 행동에 그대로 드러나면, 조작 없이도 "내가 한 일의 결과를 본다"는 자기결정성(유능감)이 성립함
- **핵심 원칙**: **토큰 경제(거리 기준 획득)는 그대로 유지.** 방향 E는 토큰과 무관한 순수 연출 레이어 — 런 하나를 마친 뒤 그 런의 "성격"을 캐릭터 리액션으로 번역하는 것
- **런 태그 시스템**: 런 데이터를 조합 가능한 태그로 분류해서, 태그별로 준비한 대사/포즈를 조합 → 문장을 낱개로 다 쓰지 않아도 조합으로 다양성 확보

  | 태그 축 | 판정 기준 (초안, 튜닝 필요) | 리액션 예 |
  |---|---|---|
  | 기온 | 더움(예: 28°C+) / 보통 / 추움 | 더움 → 땀 스프라이트 + "I need ice americano" |
  | 날씨 상태 | 비 / 눈 / 맑음 / 흐림 | 비 → 젖은 모습 + "so wet..." / 눈 → "so cold... but pretty" |
  | 페이스 유형 | 느림+장거리(LSD) / 평범 / 빠름 | LSD → 지친 자세 + "long one today... 🥱" / 빠름 → "that was FAST!" |
  | 거리 | 짧음 / 보통 / 장거리 | 장거리 → 다리 후들거리는 포즈 |

  조합 예: 더움 + LSD → "so hot... and so long... 😮‍💨" + 땀+헥헥대는 자세
- **⚠️ 기술 검증 필요 (구현 착수 전 확인)**:
  1. **`Toybox.Weather`가 Forerunner 255에서 지원되는지**, 그리고 **`onActivityCompleted()` 백그라운드 컨텍스트에서 호출 가능한지** — 현재 `manifest.xml`엔 `Weather` 권한이 선언 안 돼 있음(`Background`만 있음). 안 되면 날씨 축은 접고 페이스/거리 축만으로 시작
  2. **페이스/케이던스 계산** — `Activity.Info`의 `averageCadence`, `elapsedTime`, `elapsedDistance`로 평균 페이스(분/km) 산출은 가능할 것으로 보임(API 존재 확인 완료, 실제 값 검증은 미확인). LSD 판정은 "페이스 임계값 + 거리 임계값" 둘 다 넘으면으로 단순하게 시작

- **날씨 데이터 획득 방법 조사 (2026-07-15, 공식문서+포럼 확인)**:
  - **`Toybox.Weather`(내장, 권장)** — Forerunner 255/255s 공식 지원 기기 목록에 있음. `getCurrentConditions()`는 **동기 즉시 반환**(콜백 불필요, 없으면 `null`) → 구현 단순. 워치에 이미 캐싱된 날씨를 읽는 방식으로 보여 **폰 연결과 무관**할 가능성 높음 = 러닝 중 연결 끊김 리스크 없음. **단 백그라운드 컨텍스트 호출 가능 여부는 공식 문서에 명시 없음** → 이 프로젝트가 겪은 `Notifications.showNotification()` 백그라운드 크래시와 유사 리스크. SDK에서 `onActivityCompleted()` 안에 한 줄 넣어 `System.println()`으로 결과 찍는 게 가장 빠른 검증
  - **외부 API `Communications.makeWebRequest()`(최후 수단)** — 가능은 하나 **폰(Garmin Connect Mobile) 브리지를 거침**. 워치가 인터넷 직결 안 됨 → 폰이 블루투스로 근처에 붙어 있고 GCM 앱이 살아있어야 성공, 아니면 `-104`(BLE_CONNECTION_UNAVAILABLE). HTTPS 필수(`-1001`). `manifest.xml`에 `Communications` 권한 필요. **러닝 앱 특성상 폰 없이 뛰는 유저가 많아 신뢰도 낮음** → 날씨 하나 때문에 이 의존성을 끌어들이는 건 비추. 내장 Weather가 백그라운드에서 막힐 때만 검토하고, 그때도 "폰 없으면 날씨 태그 스킵" 방어 로직 필수
  - **결론**: 내장 `Toybox.Weather` 우선. 어느 쪽이든 **날씨 태그 실패 시 페이스/거리 태그(Tier 1 완성분)는 그대로 작동**하도록 분리 설계됨 → 이 불확실성이 방향 E 전체를 막지 않음
- **구현 지점(예상)**: `GamigotchiBackground.onActivityCompleted()`에서 런 종료 시점에 태그 계산 → `Background.exit()` 데이터에 태그 포함 → `GamigotchiApp.onBackgroundData()`에서 리액션 메시지 선택·표시 (기존 `bonus` 플래그 전달 방식과 동일 패턴)

## 기술 제약 및 확인 필요 사항

### 확인 완료 ✅

- **런 중 거리 읽기** — `Activity.getActivityInfo().elapsedDistance`로 현재 달리기 거리(미터) 실시간 읽기 가능
  - 누적 히스토리 API는 없음 → 앱이 열려 있는 동안 거리를 읽고 직접 저장하는 방식으로 구현
- **데이터 저장** — `Application.Storage.setValue/getValue`로 영구 저장 가능 (앱 재시작 후에도 유지)
  - 저장 가능 타입: Number, Float, String, Boolean, Array, Dictionary 등
  - 값당 최대 32KB 제한
- **토큰 계산** — API 문제 아님. 저장된 거리를 수식으로 토큰으로 변환하면 됨 (예: 1km = N토큰)

### 핵심 데이터 흐름

```
앱 열림 (런 중) → elapsedDistance 읽기 → 런 종료 시 거리 저장
→ 거리 × 토큰 공식 = 토큰 획득 → Storage에 누적 저장
```

### Storage 데이터 구조 (게이지 시스템, 2026-07-12 구현 완료)

| 키 | 타입 | 설명 |
|---|---|---|
| `tokens` | Number | 보유 토큰 수 |
| `growthStage` | Number | 0=알, 1=아기, 2=어른 |
| `healthStatus` | Number | 0=정상, 1=아픔, 2=사망 |
| `hunger` | Float | 배고픔 게이지 (0~100) |
| `happiness` | Float | 행복 게이지 (0~100) |
| `poopCount` | Number | 쌓인 응아 개수 |
| `poopAccumSeconds` | Number | 다음 응아 발생까지 누적된 시간 |
| `healthyElapsedSeconds` | Number | 두 게이지가 기준치 이상으로 유지된 누적 시간(성장 타이머) |
| `lastTickTime` | Number | 게이지 감소량 계산용 마지막 업데이트 시각 |
| `sickSinceTime` | Number | 아픈 상태 진입 시각 (사망 판정용) |

### 필수 확인 (개발 시작 전)

- **[확인 완료 ✅] 백그라운드 런 감지** — `Background.registerForActivityCompletedEvent()`로 앱이 닫혀 있어도 런 종료 이벤트 수신 가능
  - `ServiceDelegate`를 상속한 백그라운드 서비스 클래스 구현 → `onActivityCompleted()` 메서드에서 거리 읽기 → 토큰 계산 → `Storage` 저장 → `Background.exit(data)`로 메인 앱에 전달
  - 메인 앱 열릴 때 `AppBase.onBackgroundData()`로 백그라운드 결과 수신
  - 관련 API: `Toybox.Background`, `Toybox.Activity`
- **[확인 완료 ✅] 알림 기능** — `Toybox.Notifications.showNotification()`으로 즉시 알림 표시 가능. 단, 예약 발송 기능은 없음
  - 구현 방법: 밥줄 때마다 `Background.registerForTemporalEvent(48시간 후)`로 재등록 → 시간 됐을 때 백그라운드에서 마지막 급식 시각 확인 → 조건 충족 시 `showNotification()` 호출
  - `Notifications` 퍼미션 선언 필요 (manifest.xml)
  - 주의: 앱 종류에 따라 지원 기기가 다를 수 있음. Forerunner 255 지원 여부 실기기 확인 필요

- **[확인 완료 ✅] 워치페이스 연동 불가** — Connect IQ 앱은 각자 독립된 Storage를 가지며 앱 간 데이터 공유 없음
  - 워치페이스에서 장치 앱의 Storage를 읽는 것 불가
  - **결론**: 장치 앱 단독으로 구현하는 현재 방향이 맞음. 워치페이스 연동은 고려 불필요
  - **대안으로 고려 가능**: Glance(글랜스) — 버튼 한 번으로 접근하는 미니 뷰. 별도 앱이지만 장치 앱과 함께 번들 배포 가능한지 추가 확인 필요 (v2 검토 항목)

## 배터리 / 백그라운드 (2026-07-13 분석 + 배포 전 할 일)

### 현황 — 배터리 부담은 낮은 편

- **GPS·심박·가속도 등 센서를 직접 돌리는 게 하나도 없음** → 진짜 배터리 원흉이 없어 데이터필드/러닝 앱보다 훨씬 가벼움
- 상시 드레인은 **Temporal event 하나뿐**: `GamigotchiBackground.onTemporalEvent()`가 **5분마다 스스로 재등록**(`GamigotchiBackground.mc:39`). 하는 일은 극히 가벼움(Storage 몇 개 + 게이지 산술 + 알림 체크). Connect IQ가 백그라운드 실행을 ~30초/제한 메모리로 강제 제한
- `onActivityCompleted`(런 종료 시만)·`registerForActivityCompletedEvent`(이벤트 대기, 폴링 아님)·`_animTimer`(화면 켜져 있을 때만 500ms) → 전부 무시 가능
- 유일한 지속 비용은 **CPU를 하루 288번(5분×24h) 깨우는 것 자체**. 5분은 Garmin temporal event 최소 간격이라 현재는 허용된 최대 빈도로 도는 셈

### ⚠️ 배포 전 반드시 할 일

- **테스트용 30초 이벤트 정리** — `GamigotchiApp.onStart:36`에 30초짜리 temporal event가 남아 있음(주석: "실제는 5분 간격으로 변경"). 배포 전 제거/조정 필수

### 최적화 여지 (선택, 배터리 더 아끼려면)

- 핵심: **게이지 감소(`tick`)는 "경과 시간 기반"이라 자주 안 돌아도 됨** — 다음 실행 때 밀린 만큼 한 번에 계산. 따라서 5분 폴링이 진짜 필요한 건 (1) 게이지 낮을 때 제때 알림, (2) 달리는 중 실시간 거리 추적, 둘뿐
- → **유휴 시(게이지 높고 안 뛰는 중) 간격 백오프**로 웨이크업 3~6배 절감 가능:
  - 간단히: 유휴 시 15~30분 간격으로 늘림
  - 더 똑똑하게: "게이지가 경고선에 닿을 시각"을 계산해 그때 한 번만 깨우기
- 트레이드오프: 백그라운드 빈도를 줄이면 "돌아와" 알림 타이밍이 둔해짐 → 리텐션 vs 배터리 균형은 실기기 테스트에서 조정
